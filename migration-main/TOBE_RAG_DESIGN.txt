1. RULE_VM001 적용: 2번 서브쿼리 A를 1번에 merge

RULE_VM001은 네 정의대로:

inline view가 단순 projection/join 용도면
base table join으로 풀어서
view merging 가능하게 만든다

지금 2번은 딱 그 케이스야.

2번 원본
SELECT d.department_id,
       d.department_name,
       l.street_address,
       l.postal_code
FROM departments d,
     locations l
WHERE d.location_id = l.location_id

이건

DISTINCT 없음
GROUP BY 없음
집계 없음
단순 join + projection

이라서 독립적으로 유지할 이유가 없음.

그래서 1번에서 이 부분:

FROM employee e,
     (SUBQUERY_A) dept_locs_v
WHERE dept_locs_v.department_id = e.department_id

가 이렇게 펴짐:

FROM employee e,
     departments d,
     locations l
WHERE d.location_id = l.location_id
  AND d.department_id = e.department_id

그리고 SELECT 절도 바뀜:

기존:

SELECT e.first_name,
       e.last_name,
       dept_locs_v.street_address,
       dept_locs_v.postal_code

변경:

SELECT e.first_name,
       e.last_name,
       l.street_address,
       l.postal_code

즉 RULE_VM001만 적용한 중간 결과는 이렇게 된다.

SELECT e.first_name,
       e.last_name,
       l.street_address,
       l.postal_code
FROM employee e,
     departments d,
     locations l
WHERE d.location_id = l.location_id
  AND d.department_id = e.department_id
  AND e.department_id IN (
      SELECT d2.department_id
      FROM departments d2
      WHERE d2.department_name LIKE 'S%'
  )
  AND e.last_name = 'Smith';
2. RULE_VM004 적용: 3번 서브쿼리 B를 unnest

RULE_VM004는:

IN (SELECT ...)
subquery unnesting
join 형태로 변환

지금 3번은:

SELECT d2.department_id
FROM departments d2
WHERE d2.department_name LIKE 'S%'

그리고 1번에서는 이게 이렇게 쓰였지:

e.department_id IN (SUBQUERY_B)

이걸 그대로 풀면 의미상:

e.department_id = d2.department_id
AND d2.department_name LIKE 'S%'

인 조인으로 바꿀 수 있음.

즉 VM004만 따로 적용한 형태는:

SELECT e.first_name,
       e.last_name,
       dept_locs_v.street_address,
       dept_locs_v.postal_code
FROM employee e,
     (SUBQUERY_A) dept_locs_v,
     departments d2
WHERE dept_locs_v.department_id = e.department_id
  AND e.department_id = d2.department_id
  AND d2.department_name LIKE 'S%'
  AND e.last_name = 'Smith';

이건 아직 SUBQUERY_A는 안 푼 상태고, IN만 join으로 바꾼 중간 결과라고 보면 돼.

3. 두 룰을 같이 적용하면

이제 둘을 동시에 반영하면:

(1) SUBQUERY_A 제거

dept_locs_v → departments d + locations l

(2) SUBQUERY_B 제거

IN (SELECT d2.department_id ...) → departments d2 join

그럼 일단 기계적으로는 이렇게 될 수 있어:

SELECT e.first_name,
       e.last_name,
       l.street_address,
       l.postal_code
FROM employee e,
     departments d,
     locations l,
     departments d2
WHERE d.location_id = l.location_id
  AND d.department_id = e.department_id
  AND e.department_id = d2.department_id
  AND d2.department_name LIKE 'S%'
  AND e.last_name = 'Smith';
4. 그런데 여기서 한 번 더 정리해야 함

왜냐하면 d와 d2가 사실상 같은 departments를 보고 있기 때문이야.

d.department_id = e.department_id
d2.department_id = e.department_id

이면 결국

d.department_id = d2.department_id

가 되니까, 둘을 따로 둘 이유가 약함.

그리고 원래 서브쿼리 A에서 이미 department_name도 가져오고 있었기 때문에,
서브쿼리 B의 조건 department_name LIKE 'S%'는 사실 d 쪽으로 합칠 수 있어.

즉:

d.department_name LIKE 'S%'

를 쓰면 충분함.

그래서 최종 통합 결과는 이렇게 정리된다.

SELECT e.first_name,
       e.last_name,

     locations l
WHERE d.location_id = l.location_id
  AND d.department_id = e.department_id
  AND d.department_name LIKE 'S%'
  AND e.last_name = 'Smith';
